[#introduction]
== Introduction

As detailed in the link:ztp-workflow.html#abi-deployment[Agent Based Installation] section, this installation method combines the ease of use of the Assisted Installation service with the ability to run offline. It also optionally generates or accepts link:ztp-intro.html[Zero Touch Provisioning] (ZTP) custom resources. In this part of the lab, we will:

* Create the prerequisites for the OpenShift cluster installation.
* Define the OpenShift cluster as a `ClusterInstance` CR.
* Include any extra configuration required for our telco 5G Core-specific use case.
* Commit all manifests into the `ztp-repository` Git repo, where the link:ztp-intro.html[OpenShift GitOps] operator automatically applies the changes.

[#baremetal-node-details]
== Bare Metal Node Details

The details for our bare-metal nodes, which we want to provision as OpenShift Nodes, are as follows:

* RedFish Endpoint: `redfish-virtualmedia://192.168.125.1:9000/redfish/v1/Systems/local/ocp-mno-<master/worker>-<node-number>`
* MAC Address: `aa:aa:aa:aa:XX:YY`
* Primary disk: `/dev/vda`
* BMC User: `admin`
* BMC Password: `admin`

[#reference-manifest-clusterinstance]
== Reference Manifest Configuration

Let's add the {rds-config}[reference Core configuration] for the version of OCP we are about to deploy. In the workstation, run the following command to extract the reference ZTP {lab-major-version} Core configuration. These manifests are extracted from the `ztp-site-generate` container.

[.console-input]
[source,bash,subs="attributes+,+macros"]
-----
podman login infra.5g-deployment.lab:8443 -u admin -p r3dh4t1! --tls-verify=false
podman run --log-driver=none --rm --tls-verify=false {ztp-core-reference-ocp-release-disconnected-image} | base64 -d | tar x -C ~/5g-deployment-lab/ztp-repository/site-configs/install-manifests/{lab-minor-version}/extra-manifests/ --strip-components=3 telco-core-rds/install/extra-manifests/
podman run --log-driver=none --rm --tls-verify=false {ztp-core-reference-ocp-intermediate-release-disconnected-image} | base64 -d | tar x -C ~/5g-deployment-lab/ztp-repository/site-configs/install-manifests/{lab-mid-version}/extra-manifests/ --strip-components=3 telco-core-rds/install/extra-manifests/
podman run --log-driver=none --rm --tls-verify=false {ztp-core-reference-ocp-next-release-disconnected-image} | base64 -d | tar x -C ~/5g-deployment-lab/ztp-repository/site-configs/install-manifests/{lab-major-version}/extra-manifests/ --strip-components=3 telco-core-rds/install/extra-manifests/
-----

`reference-manifest/{lab-major-version}/` refers to the Git subdirectory that stores the contents of the Core configuration for a specific version, in this case {lab-major-version}. Different versions of the Core configuration can coexist by extracting them from the appropriate `ztp-site-generate` container version to a different `extra-manifest` folder. Then, the user only needs to include the right `reference-manifest` version folder to the cluster version being installed.

This configuration is applied during installation time. 

Our recommendation is to always create a reference and keep the reference and user defined manifests (see next section) separately. This makes later updates to the reference based on z-stream releases significantly easier (simply replace the contents of reference-manifests with updated content)
