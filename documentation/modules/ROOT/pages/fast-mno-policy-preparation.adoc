[#introduction]
== Introduction

In this section you will perform a fully automated EUS-to-EUS upgrade of the `mno` managed cluster from *{lab-minor-version}* to *{lab-major-version}*. The cluster has 3 worker MachineConfigPools (`worker-0`, `worker-1`, `worker-2`), each containing 3 nodes in its own Kubernetes failure domain.

Following the ZTP GitOps methodology used throughout this lab, the ACM upgrade policies are defined as xref:policygen-deepdive.adoc#policy-generator[PolicyGenerator] CRs that reference individual Kubernetes resource files (reference CRs). These files are committed to the ZTP Git repository and ArgoCD automatically deploys the resulting RHACM policies to the Hub cluster. The `ClusterGroupUpgrade` (CGU) resources that orchestrate the upgrade stages are the only objects applied directly to the Hub cluster.

You will:

. Verify that the cluster and Hub components are in the correct starting state.
. Create the reference CRs and PolicyGenerator files for the {lab-minor-version} → {lab-mid-version} upgrade, commit them to the ZTP repository, and confirm ArgoCD deploys them.
. Create the reference CRs and PolicyGenerator files for the {lab-mid-version} → {lab-major-version} upgrade and worker zone policies in the same way.
. Define and apply a chain of `ClusterGroupUpgrade` (CGU) resources to orchestrate the full upgrade sequence.
. Trigger the upgrade and monitor it as it progresses through each stage automatically.

Before starting, make sure you have read and understood the xref:fast-mno-upgrades.adoc[Fast MNO Upgrades — Theory] section. All policies and CGUs in this lab are a direct application of the three-pillar methodology described there.

== Setting Up the Reference CRs Directory

In the ZTP GitOps workflow, ACM policies are not written by hand as raw Policy YAMLs. Instead, we use a pre-built container image that contains small, focused *reference CR* files that describe the desired Kubernetes objects, and then declare a `PolicyGenerator` CR that wraps them into ACM Policies automatically. ArgoCD watches the ZTP Git repository and pushes any new or updated policies to the Hub cluster. Refer to the xref:rhacm-policies.adoc#policy-framework[RHACM Governance Policy Framework] and xref:policygen-deepdive.adoc#policy-generator[PolicyGenerator Deep Dive] sections for background.

[#automation-with-acm-and-talm]
== Automation with ACM and TALM

The entire end-to-end procedure is orchestrated using Red Hat Advanced Cluster Management (ACM) and the Topology Aware Lifecycle Manager (TALM) operator.

* **Policies**: ACM Policies are created on the Hub cluster for each distinct upgrade action (e.g., core-upgrade-ocp-{version-suffix-mid}, core-upgrade-olm-{version-suffix-mid}, core-upgrade-ocp-{version-suffix-major}, core-upgrade-workers-0, etc.). These policies are initially set to remediationAction: inform.
* **Orchestration**: A series of ClusterGroupUpgrade (CGU) custom resources are used to trigger and manage these policies in the correct sequence. TALM is responsible for moving policies from an 'inform' state to an 'enforce' state.
* **Sequencing**: The blockingCRs stanza within a CGU is used to create dependencies, ensuring that one stage (e.g., the upgrade to {lab-major-version}) does not begin until the prerequisite stage (e.g., the upgrade to {lab-mid-version}) has successfully completed.

This approach transforms a complex, high-risk manual upgrade into a predictable, repeatable, and fully automated _zero-touch_ operation that executes significantly faster than traditional methods.


[#policies-19]
== Creating the {lab-minor-version} → {lab-mid-version} Upgrade Policies

The core of the automation is a set of ACM Policies that declaratively define the desired state for each step of the upgrade. These policies are applied to the Hub cluster and remain in an inform state until activated by a ClusterGroupUpgrade (CGU).

The following policies are created to manage the upgrade from version  {openshift-release} to {openshift-intermediate-release}.

In this section we are going to present parts of the PolicyGenerator file that will be used to start the Fast MNO Upgrade. At the end of this section you will create the complete PolicyGenerator file based on the reference core-upgrade PolicyGenerator available in the telco-reference repository https://github.com/openshift-kni/telco-reference/blob/release-{lab-major-version}/telco-core/configuration/core-upgrade.yaml[here].

[#policy-core-upgrade-prep-mid]
=== Policy: core-upgrade-prep-{version-suffix-mid}

This is the *preparation policy*. It readies the spoke cluster for the control plane upgrade by:

* *Updating the CatalogSource*: It points the redhat-operators-disconnected catalog to the {openshift-intermediate-release} index image. This ensures OLM can find the correct operator versions compatible with {lab-mid-version}.
* *Pausing and Configuring Worker MCPs*: It patches all worker MachineConfigPools (worker-0 through worker-3) to set paused: true. This is a critical safety measure to *prevent worker nodes from upgrading while the control plane is in motion*. It also pro-actively sets maxUnavailable: "100%" in preparation for the parallel worker upgrade phase later.

[source,yaml,subs="attributes+,+macros"]
----
policies:
  - name: core-upgrade-prep-{version-suffix-mid}
    policyAnnotations:
      ran.openshift.io/soak-seconds: "30"
    manifests:
      - path: {openshift-intermediate-release}/source-crs/reference-crs/required/other/catalog-source.yaml
        patches:
        - spec:
            image: infra.5g-deployment.lab:8443/redhat/redhat-operator-index:{catalogsource-index-image-tag-ocp-intermediate-release}
        - status:
            connectionState:
              lastObservedState: READY
      # Provide the required acknowledgement of the upgrade. This
      # assumes that the verification has been done during planning
      # lab/certification phase of project
      # - path: reference-crs/optional/other/upgrade-ack.yaml
      #   patches:
      #   - data:
      #       ack-4.15-kube-1.29-api-removals-in-4.16: "true"
      - path: {openshift-intermediate-release}/source-crs/reference-crs/custom-manifests/mcp-worker-0.yaml
        patches:
        - spec:
          # Adjust as needed based on spare capacity, cluster design, and application requirements
            maxUnavailable: "100%"
            paused: true
      - path: {openshift-intermediate-release}/source-crs/reference-crs/custom-manifests/mcp-worker-1.yaml
        patches:
        - spec:
          # Adjust as needed based on spare capacity, cluster design, and application requirements
            maxUnavailable: "100%"
            paused: true
      - path: {openshift-intermediate-release}/source-crs/reference-crs/custom-manifests/mcp-worker-2.yaml
        patches:
        - spec:
          # Adjust as needed based on spare capacity, cluster design, and application requirements
            maxUnavailable: "100%"
            paused: true
----

[#policy-core-upgrade-ocp-mid]
=== Policy: core-upgrade-ocp-{version-suffix-mid}

This is the main upgrade policy. Its sole purpose is to target the ClusterVersion resource on the spoke cluster and instruct it to begin the upgrade. It sets the desired channel to stable-{lab-mid-version} and provides the specific image and version ({mno-cluster-version2}) for the cluster to upgrade to. 

To obtain the correct `image` digest for your mirror registry, run:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
podman login infra.5g-deployment.lab:8443 -u admin -p r3dh4t1! --tls-verify=false --authfile ~/.docker/config.json
oc adm release info --registry-config ~/.docker/config.json {mno-cluster-version2-cvo} | grep 'Pull From: '
----

Replace the `image` value below with the output of that command.

[source,yaml,subs="attributes+,+macros"]
----
policies:
  - name: core-upgrade-ocp-{version-suffix-mid}
    manifests:
      - path: {openshift-intermediate-release}/source-crs/reference-crs/optional/other/ClusterVersion.yaml
        patches:
        - spec:
            channel: stable-{lab-mid-version}
            desiredUpdate:
              # Replace version, image and status below with desired version
              version: {mno-cluster-version2-cvo}
              image: infra.5g-deployment.lab:8443/openshift/release-images@{fast-upgrade-ocp-intermediate-image-digest}
              force: true
        - status:
            history:
            - version: {mno-cluster-version2-cvo}
              state: "Completed"
----

[#worker-upgrade-policies]
== Defining the Worker Upgrade Policies

With the control plane policies ready to be upgraded to {lab-major-version}, the worker nodes are still paused. This final stage orchestrates the sequential, parallel upgrade of each worker MachineConfigPool (MCP).

Next, we create a separate policy for each worker MCP. The sole purpose of each policy is to unpause its corresponding MCP by setting paused: false. The MCO on the spoke cluster will see this change and—because we set maxUnavailable: "100%" in the preparation policy—will begin upgrading all nodes in that MCP simultaneously.

Here is the policy for worker-0. This manifest is duplicated for worker-1, worker-2, and worker-3.

== Placement and Binding

The PlacementRule and PlacementBinding are included into the PolicyGenerator CRs to apply all the policies related to the upgrade to {lab-major-version}, e.g cdocumentation/modules/ROOT/pages/installing-agentbasedinstall-cluster.adocontrol-plane and worker policies. 

> IMPORTANT: The placementRule is configured to select all clusters with the following labels: common set to {policygen-common-label-ocp-release}, {policygen-common-label-ocp-intermediate-release}, rds set to core, and the existence of the {fast-upgrade-label} label.
