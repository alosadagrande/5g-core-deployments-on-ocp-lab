[#introduction]
== Introduction

In this section you will perform a fully automated EUS-to-EUS upgrade of the `mno` managed cluster from *{lab-minor-version}* to *{lab-major-version}*. The cluster has 3 worker MachineConfigPools (`worker-0`, `worker-1`, `worker-2`), each containing 3 nodes in its own Kubernetes failure domain.

Following the ZTP GitOps methodology used throughout this lab, the ACM upgrade policies are defined as xref:policygen-deepdive.adoc#policy-generator[PolicyGenerator] CRs that reference individual Kubernetes resource files (reference CRs). These files are committed to the ZTP Git repository and ArgoCD automatically deploys the resulting RHACM policies to the Hub cluster. The `ClusterGroupUpgrade` (CGU) resources that orchestrate the upgrade stages are the only objects applied directly to the Hub cluster.

You will:

. Verify that the cluster and Hub components are in the correct starting state.
. Create the reference CRs and PolicyGenerator files for the {lab-minor-version} → {lab-mid-version} upgrade, commit them to the ZTP repository, and confirm ArgoCD deploys them.
. Create the reference CRs and PolicyGenerator files for the {lab-mid-version} → {lab-major-version} upgrade and worker zone policies in the same way.
. Define and apply a chain of `ClusterGroupUpgrade` (CGU) resources to orchestrate the full upgrade sequence.
. Trigger the upgrade and monitor it as it progresses through each stage automatically.

Before starting, make sure you have read and understood the xref:fast-mno-upgrades.adoc[Fast MNO Upgrades — Theory] section. All policies and CGUs in this lab are a direct application of the three-pillar methodology described there.

[#policies-19]
== Creating the {lab-minor-version} → {lab-mid-version} Upgrade Policies

In the ZTP GitOps workflow, ACM policies are not written by hand as raw Policy YAMLs. Instead, you create small, focused *reference CR* files that describe the desired Kubernetes objects, and then declare a `PolicyGenerator` CR that wraps them into ACM Policies automatically. ArgoCD watches the ZTP Git repository and pushes any new or updated policies to the Hub cluster. Refer to the xref:rhacm-policies.adoc#policy-framework[RHACM Governance Policy Framework] and xref:policygen-deepdive.adoc#policy-generator[PolicyGenerator Deep Dive] sections for background.

All upgrade policies are placed in the `ztp-policies` namespace on the Hub cluster (controlled by the `policyDefaults.namespace` field in the PolicyGenerator CR).

[#ref-crs-19]
=== Creating the Reference CRs for {lab-minor-version} → {lab-mid-version}

Reference CRs are plain Kubernetes resource files. The PolicyGenerator plugin reads them and embeds them as policy object-templates. Create each file below inside `source-crs/`.

[#ref-catalogsource-19]
==== CatalogSource — {lab-mid-version} operator index

This reference CR updates the disconnected operator catalog to point to the `{lab-mid-version}` index image. Policy `core-upgrade-prep-19` will enforce it on the spoke cluster before the upgrade starts.

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF > source-crs/CatalogSource-419.yaml
apiVersion: operators.coreos.com/v1alpha1
kind: CatalogSource
metadata:
  name: redhat-operators-disconnected
  namespace: openshift-marketplace
spec:
  displayName: Red Hat Disconnected Operators Catalog
  image: registry.redhat.io/redhat/redhat-operator-index:v{lab-mid-version}
  publisher: Red Hat
  sourceType: grpc
status:
  connectionState:
    lastObservedState: READY
EOF
----

[#ref-clusterversion-19]
==== ClusterVersion — target {lab-mid-version}

This reference CR targets the intermediate EUS version. The `image` field must reference the release image available in your disconnected registry.

[NOTE]
====
To obtain the correct `image` digest for your mirror registry, run:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
oc adm release info {lab-mid-version} --registry-config ~/pull-secret.json \
  | grep 'Pull From'
----

Replace the `image` value below with the output of that command.
====

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF > source-crs/ClusterVersion-419.yaml
apiVersion: config.openshift.io/v1
kind: ClusterVersion
metadata:
  name: version
spec:
  channel: eus-{lab-mid-version}
  desiredUpdate:
    force: false
    image: <mirror-registry>/openshift-release-dev/ocp-release:{lab-mid-version}-x86_64
    version: {lab-mid-version}
  upstream: ""
EOF
----

[#ref-mcp-paused-19]
==== MachineConfigPools — paused with maxUnavailable 100%

These three reference CRs ensure all worker MCPs are paused and configured for a parallel (all-at-once) reboot when eventually unpaused.

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
for MCP in worker-0 worker-1 worker-2; do
  cat <<EOF > source-crs/MachineConfigPool-${MCP}-paused.yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfigPool
metadata:
  labels:
    machineconfiguration.openshift.io/role: ${MCP}
  name: ${MCP}
spec:
  maxUnavailable: "100%"
  paused: true
EOF
done
----

[#ref-validate-19]
==== ValidateSubscriptions — compliance check after OLM upgrades

This reference CR is a `ConfigurationPolicy` that uses an `object-templates-raw` block to verify that all operator Subscriptions have reconciled after the upgrade. The annotation `ztp-validated: "4.19"` is used by TALM's soak check mechanism.

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF > source-crs/ValidateSubscriptions.yaml
apiVersion: policy.open-cluster-management.io/v1
kind: ConfigurationPolicy
metadata:
  name: validate-subscriptions
spec:
  remediationAction: inform
  severity: low
  object-templates-raw: |
    {{- range (lookup "operators.coreos.com/v1alpha1" "Subscription" "" "").items }}
    - complianceType: musthave
      objectDefinition:
        apiVersion: operators.coreos.com/v1alpha1
        kind: Subscription
        metadata:
          name: {{ .metadata.name }}
          namespace: {{ .metadata.namespace }}
          annotations:
            ztp-validated: "4.19"
        status:
          state: AtLatestKnown
    {{- end }}
EOF
----

[#policygen-19]
=== Creating the PolicyGenerator for {lab-minor-version} → {lab-mid-version}

Now create the `PolicyGenerator` CR that bundles the reference CRs created above into four ACM Policies. The `placement.labelSelector` block defines which managed clusters receive the policies — TALM will add and remove these labels automatically as the upgrade progresses.

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF > upgrade-419.yaml
apiVersion: policy.open-cluster-management.io/v1
kind: PolicyGenerator
metadata:
  name: upgrade-mno-19
placementBindingDefaults:
  name: upgrade-19-placement-binding
policyDefaults:
  namespace: ztp-policies
  placement:
    labelSelector:
      matchExpressions:
        - key: common
          operator: In
          values:
            - ocp{lab-minor-version-nodots}
        - key: rds
          operator: In
          values:
            - core
        - key: upgrade-version-4-19
          operator: Exists
  remediationAction: inform
  severity: low
policies:
- name: core-upgrade-prep-19
  policyAnnotations:
    ran.openshift.io/soak-seconds: "30"
  manifests:
    - path: source-crs/CatalogSource-419.yaml
    - path: source-crs/MachineConfigPool-worker-0-paused.yaml
    - path: source-crs/MachineConfigPool-worker-1-paused.yaml
    - path: source-crs/MachineConfigPool-worker-2-paused.yaml
- name: core-upgrade-ocp-19
  manifests:
    - path: source-crs/ClusterVersion-419.yaml
- name: core-upgrade-olm-19
  policyAnnotations:
    ran.openshift.io/soak-seconds: "60"
  manifests:
    - path: source-crs/Subscription-cluster-logging-419.yaml
    - path: source-crs/Subscription-sriov-419.yaml
    - path: source-crs/Subscription-odf-419.yaml
    - path: source-crs/Subscription-metallb-419.yaml
    - path: source-crs/Subscription-numaresources-419.yaml
- name: core-upgrade-validate-19
  policyAnnotations:
    ran.openshift.io/soak-seconds: "30"
  manifests:
    - path: source-crs/ValidateSubscriptions.yaml
      patches:
        - metadata:
            annotations:
              ztp-validated: "4.19"
EOF
----

[#cgus]
== Defining the CGU Orchestration Chain

With all policies in place, the next step is to define the three `ClusterGroupUpgrade` (CGU) resources that orchestrate the entire upgrade sequence. For a detailed explanation of how TALM and CGUs work, refer to the xref:talm.adoc#talm[Topology Aware Lifecycle Manager (TALM)] section.

The key mechanism is the `blockingCRs` field: a CGU will remain in a waiting state until all CGUs listed in its `blockingCRs` have completed successfully. By applying all three CGUs simultaneously, TALM automatically manages the sequencing — you do not need to watch and manually trigger each stage.

The `actions.afterCompletion` block in each CGU controls cluster labeling: it removes the labels that activated the current stage and adds the labels needed for the next stage, ensuring the correct policies are activated at the right time.

[#cgu-19]
=== CGU: upgrade-19

Manages the full control plane upgrade from `{lab-minor-version}` to `{lab-mid-version}`. When it completes, it automatically:

* Removes the `upgrade-version-4-19` label (deactivating the `{lab-mid-version}` placement).
* Adds `common=ocp419` and `upgrade-version-4-20` (activating the `{lab-major-version}` placement and unblocking `upgrade-20`).

Create `cgu-upgrade-19.yaml`:

[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: upgrade-19
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      deleteClusterLabels:
        upgrade-version-4-19: ""
      addClusterLabels:
        common: "ocp419"
        upgrade-version-4-20: ""
      deleteObjects: true
  clusters:
  - mno
  enable: true
  managedPolicies:
  - core-upgrade-prep-19
  - core-upgrade-ocp-19
  - core-upgrade-olm-19
  - core-upgrade-validate-19
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
----

[#cgu-20]
=== CGU: upgrade-20

Manages the full control plane upgrade from `{lab-mid-version}` to `{lab-major-version}`. It is blocked by `upgrade-19` and, when it completes, adds the `control-plane=4.20` label to signal that the control plane is fully upgraded.

Create `cgu-upgrade-20.yaml`:

[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: upgrade-20
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      deleteObjects: true
      addClusterLabels:
        control-plane: "{lab-major-version}"
  blockingCRs:
  - name: upgrade-19
    namespace: ztp-policies
  clusters:
  - mno
  enable: true
  managedPolicies:
  - core-upgrade-prep-20
  - core-upgrade-ocp-20
  - core-upgrade-olm-20
  - core-upgrade-validate-20
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
----

[#cgu-finish]
=== CGU: upgrade-finish

Manages the sequential, parallel upgrade of each of the 3 worker zones. It is blocked by `upgrade-20`. `maxConcurrency: 1` ensures that TALM enforces the zone policies one at a time: `core-upgrade-workers-0` is enforced first, and only once `worker-0` MCP reports `Updated: True` does TALM move to `core-upgrade-workers-1`, then `core-upgrade-workers-2`.

Create `cgu-upgrade-finish.yaml`:

[source,yaml]
----
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: upgrade-finish
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      addClusterLabels:
        common: "ocp420"
        version: "{lab-major-version}"
      deleteClusterLabels:
        upgrade-version-4-20: ""
        control-plane: "{lab-major-version}"
  blockingCRs:
  - name: upgrade-20
    namespace: ztp-policies
  clusters:
  - mno
  enable: true
  managedPolicies:
  - core-upgrade-workers-0
  - core-upgrade-workers-1
  - core-upgrade-workers-2
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
----
