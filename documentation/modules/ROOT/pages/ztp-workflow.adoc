= ZTP Workflow
include::_attributes.adoc[]
:profile: 5g-core-lab

As mentioned, Zero Touch Provisioning (ZTP) leverages multiple products or link:ztp-at-scale.html[components] to deploy OpenShift Container Platform clusters using a GitOps approach. While the workflow starts when the site is connected to the network and ends with the CNF workload deployed and running on the site nodes, it can be logically divided into two stages: provisioning of the MNO cluster and applying the desired configuration, which in our case is applying the {core-rds-link}[Telco Core reference configuration].

IMPORTANT: The workflow requires no manual intervention; ZTP automatically configures the MNO cluster once it is provisioned.

[#deployment-methods]
== Multi-Node OpenShift Deployments

For Telco Core, ZTP provisions multi-node OpenShift (MNO) clusters using the *Agent Based Installation* flow through the link:ztp-at-scale.html#siteconfig[SiteConfig Operator]. The link:ztp-at-scale.html#ai[Assisted Service] is central to this deployment method.

It starts with creating declarative configurations for the provisioning of your OpenShift clusters. This manifest is defined in a custom resource called `ClusterInstance`. In a disconnected environment, a container registry is needed to deliver the OpenShift container images required for the installation. This can be achieved using {disconnected-registry-docs}[oc-mirror] (starting in OCP 4.18, oc-mirror v2 is the preferred approach). More information about disconnected environments can be found in link:deployment-considerations.html#disconnected-environments[Deployment Considerations section].

NOTE: A Telco Core MNO cluster requires a highly available control plane consisting of three or more nodes, plus one or more worker nodes. Worker nodes can be further subdivided into groups using `MachineConfigPool` custom resources to support different hardware profiles or upgrade strategies.

WARNING: Depending on your specific environment, you might need extra services such as DHCP, DNS, NTP, or HTTP. The latter is needed for downloading the RHCOS live ISO and the RootFS image locally instead of using the default http://mirror.openshift.com site.

Once the configuration is created, you can push it to the Git repository where Argo CD continuously monitors for new content:

image::ztp_workflow_0.png[ZTP Workflow 0]

Argo CD pulls the ClusterInstance CR, which the link:ztp-at-scale.html#siteconfig[SiteConfig Operator] validates and transforms into several custom resources that are understood by the link:ztp-at-scale.html#ai[Assisted Service] running on the hub cluster. A ClusterInstance contains all the necessary information to provision your cluster nodes. It creates ISO images with the defined configuration that are delivered to the bare-metal nodes to begin the installation process. These images are used to repeatedly provision large numbers of nodes efficiently and quickly, allowing you to scale Core cluster deployments across your infrastructure.

IMPORTANT: In telco use cases, clusters primarily run on bare-metal hosts. Therefore, the produced ISO images are mounted using remote virtual media features of the baseboard management controller (BMC).

image::ztp_workflow_1.png[ZTP Workflow 1]

The resulting resources depicted in the previous image are called Installation Manifests. For this workflow, they are:

* AgentClusterInstall
* ClusterDeployment
* NMStateConfig
* KlusterletAddonConfig
* ManagedCluster
* InfraEnv
* BareMetalHost
* HostFirmwareSettings
* ConfigMap for extra-manifest configurations

Then, they are processed by the Assisted Service, and the provisioning process starts.

image::ztp_workflow_2.png[ZTP Workflow 2]

The provisioning process includes installing the host operating system (RHCOS) on a blank server and deploying OpenShift Container Platform. This stage is managed mainly by the Assisted Service, which is part of the Infrastructure Operator. Detailed information about the workflow controlled by this software is available in the previous link:ztp-at-scale.html#ai[Infrastructure operator] section.

IMPORTANT: ZTP allows provisioning clusters at scale. Multiple ClusterInstance CRs can be committed to Git simultaneously or over time to deploy multiple MNO clusters.

[#ztp-policies]
== Deployment Configuration

Once the clusters are provisioned, the day-2 configuration defined in one or multiple `PolicyGenerator` custom resources will be automatically applied. The `PolicyGenerator` custom resource is interpreted by the ZTP pipeline using a specific kustomize plugin called {cnf-features-policygen-plugin}[policy-generator]. These templates generate Policy CRs on the hub cluster, which define the configuration to be applied to the deployed clusters. A Policy CR can be bound to multiple clusters, allowing a scalable means to define configuration across a large fleet of clusters. In Telco Core clusters, these policies configure subscriptions for day-two operators and necessary platform-level configurations such as MachineConfigPools, networking operators (SR-IOV, MetalLB, NMState), storage (ODF), and NUMA-aware workload scheduling.

image::ztp_workflow_3.png[ZTP Workflow 3]

If you want to apply a new configuration or replace an existing configuration later, you can update the PolicyGenerator in Git, which will automatically propagate to the Policy CRs on the hub cluster. After that, the user decides when those updated policies are applied using TALM.

[#takeaways]
== Takeaways

In summary, the deployment of the clusters includes:

* Leveraging a GitOps methodology for a scalable, traceable, and reliable deployment model.
* Installing the host operating system (RHCOS) on blank servers.
* Deploying OpenShift Container Platform as a highly available multi-node cluster.
* Creating cluster policies via `PolicyGenerator` and site subscriptions via `ClusterInstance`.
* Planning and applying `MachineConfigPool` labels to worker nodes for controlled upgrade windows and hardware segmentation.
* Making the necessary network configurations to the cluster operating system using the NMState Operator.
* Deploying day-two operators and performing platform-level configurations, such as SR-IOV, MetalLB, ODF, and NUMA-aware workload scheduling.
* Downloading images needed to run workloads (CNFs).
