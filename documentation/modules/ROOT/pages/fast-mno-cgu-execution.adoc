[#introduction]
== Executing the Full EUS-to-EUS Upgrade

With all policies created and bound on the Hub cluster, we are ready to orchestrate the entire, multi-stage upgrade. This is accomplished by defining and applying a series of **ClusterGroupUpgrade (CGU)** resources.

The core strategy is to **chain the CGUs** together using the blockingCRs (Blocking Custom Resources) directive. This allows us to apply all the CGUs at once, and TALM will automatically execute each major stage in the correct sequence, waiting for the previous one to complete before starting the next.

[#cgus]
== Defining the CGU Orchestration Chain

We will create four CGU files to manage the entire upgrade flow from *{lab-minor-version}* to *{lab-major-version}*.

The key mechanism is the `blockingCRs` field: a CGU will remain in a waiting state until all CGUs listed in its `blockingCRs` have completed successfully. By applying all three CGUs simultaneously, TALM automatically manages the sequencing — you do not need to watch and manually trigger each stage.

The `actions.afterCompletion` block in each CGU controls cluster labeling: it removes the labels that activated the current stage and adds the labels needed for the next stage, ensuring the correct policies are activated at the right time.

[#cgu-mid]
=== CGU: fast-upgrade-{version-suffix-mid}

This is the first CGU in the main upgrade chain. It orchestrates the prep, ocp, olm, and validate policies required to get the control plane from {lab-minor-version} to {lab-mid-version}. 

Create `fast-upgrade-{version-suffix-mid}.yaml`:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF | oc --kubeconfig ~/hub-kubeconfig apply -f -
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: fast-upgrade-{version-suffix-mid}
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      deleteObjects: true
      addClusterLabels:
        {upgrading-to-mid-label}: ""
        common: "{policygen-common-label-ocp-intermediate-release}"
  clusters:
  - mno
  enable: true
  managedPolicies:
  - core-upgrade-prep-{version-suffix-mid}
  - core-upgrade-ocp-{version-suffix-mid}
  - core-upgrade-olm-{version-suffix-mid}
  - core-upgrade-validate-{version-suffix-mid}
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
EOF
----

[#cgu-major]
=== CGU: fast-upgrade-{version-suffix-major}

This CGU manages the upgrade from {lab-mid-version} to {lab-major-version}. It is blocked by `fast-upgrade-{version-suffix-mid}` CGU and, when it completes, it automatically:

* Removes the `{upgrading-to-mid-label}` label.
* Adds `common={policygen-common-label-ocp-next-release}` and `{upgrading-to-major-label}` label.
* Cleans up the objects created by the `fast-upgrade-{version-suffix-mid}` CGU.

Create `fast-upgrade-{version-suffix-major}.yaml`:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF | oc --kubeconfig ~/hub-kubeconfig apply -f -
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: fast-upgrade-{version-suffix-major}
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      deleteClusterLabels:
        {upgrading-to-mid-label}: ""
      addClusterLabels:
        {upgrading-to-major-label}: ""
      deleteObjects: true
  clusters:
  - mno
  enable: true
  managedPolicies:
    - core-upgrade-prep-{version-suffix-major}
    - core-upgrade-ocp-{version-suffix-major}
    - core-upgrade-olm-{version-suffix-major}
    - core-upgrade-validate-{version-suffix-major}
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
  blockingCRs:
  - name: fast-upgrade-{version-suffix-mid}
    namespace: ztp-policies
EOF
----

[#cgu-workers]
=== CGU: fast-upgrade-workers

This is the final CGU in the chain. It manages the upgrade of the 3 worker zones. It is blocked by `fast-upgrade-{version-suffix-major}` and, when it completes, it automatically:

* Adds `version={lab-major-version}` label.
* Cleans up the objects created by the `fast-upgrade-{version-suffix-major}` CGU.

Create `fast-upgrade-finish.yaml`:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
cat <<EOF | oc --kubeconfig ~/hub-kubeconfig apply -f -
---
apiVersion: ran.openshift.io/v1alpha1
kind: ClusterGroupUpgrade
metadata:
  name: fast-upgrade-finish
  namespace: ztp-policies
spec:
  actions:
    afterCompletion:
      addClusterLabels:
        common: "{policygen-common-label-ocp-next-release}"
      deleteClusterLabels:
        {upgrading-to-major-label}: ""
        {fast-upgrade-label}: ""
      deleteObjects: true
  clusters:
  - mno
  enable: true
  managedPolicies:
    # These policies unpause the MCP
    - core-upgrade-workers-0
    - core-upgrade-workers-1
    - core-upgrade-workers-2
  remediationStrategy:
    maxConcurrency: 1
    timeout: 480
  blockingCRs:
  - name: fast-upgrade-{version-suffix-major}
    namespace: ztp-policies
EOF
----

[#trigger-upgrade]
== Triggering the Upgrade

After applying all three CGUs to the Hub cluster, TALM will automatically determine the correct execution order based on the `blockingCRs` dependencies and start the upgrade process.

You can check the status of all three CGUs:

[.console-input]
[source,bash,subs="attributes+,+macros"]
----
oc --kubeconfig ~/hub-kubeconfig get cgu -n ztp-policies
----

[source,console,subs="attributes+"]
----
NAME                  AGE   STATE                  DETAILS
fast-upgrade-{version-suffix-mid}       43s   InProgress             Remediating non-compliant policies
fast-upgrade-{version-suffix-major}       38s   IncompleteBlockingCR   Blocking CRs that are not completed: [fast-upgrade-{version-suffix-mid}]
fast-upgrade-finish   3s    IncompleteBlockingCR   Blocking CRs that are not completed: [fast-upgrade-{version-suffix-major}]
----

WARNING: The `IncompleteBlockingCR` state for `fast-upgrade-{version-suffix-major}` and `fast-upgrade-finish` is expected and not an error. The `fast-upgrade-{version-suffix-major}` CGU is blocked by `fast-upgrade-{version-suffix-mid}` CGU and the `fast-upgrade-finish` CGU is blocked by `fast-upgrade-{version-suffix-major}` CGU.

The upgrade is now fully automated. TALM will execute each stage in sequence without any further manual intervention.
