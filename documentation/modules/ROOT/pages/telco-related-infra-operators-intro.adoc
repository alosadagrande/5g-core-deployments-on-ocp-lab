= Introduction to Telco Core Infrastructure Operators

When working on Telco Core environments there are a set of operators that will be common to almost all of them. In this section we will introduce the various OpenShift Container Platform components that you use to configure and deploy clusters to run telco core workloads, including control plane functions (signaling, aggregation, session border controller) and centralized data plane functions such as the 5G User Plane Function (UPF). You can read the full {core-rds-link}[Telco Core Reference Design Specification (RDS)] for details.

[#node-tuning-operator]
== Node Tuning Operator

The Node Tuning Operator helps you manage node-level tuning by orchestrating the TuneD daemon and other OS-level tuning. In the context of Telco Core deployments, it is used to configure CPU partitioning and performance tuning through the Performance Profile controller.

Unlike RAN deployments, Telco Core clusters run the stock (non-realtime) kernel. The `PerformanceProfile` API is used to:

* Partition CPUs into reserved (for OS and platform overhead) and isolated (for high-performance workload) sets.
* Enable Guaranteed QoS scheduling for latency-sensitive CNFs such as DPDK-based UPFs.
* Configure IRQ load balancing — `globallyDisableIrqLoadBalancing` is set to `false` so that IRQ balancing remains globally enabled, while individual pods annotate CPUs for isolation.
* Configure huge pages for DPDK applications that require pre-allocated large memory pages.

The Node Tuning Operator is part of the standard OpenShift installation.

You can read more about the NTO {nto-docs}[here].

[#sriov-operator]
== SR-IOV Operator

This Operator helps you manage SR-IOV network devices and network attachments. SR-IOV allows a single physical NIC (Physical Function) to be divided into multiple Virtual Functions (VFs), each of which can be assigned directly to a pod for near-native network performance.

In Telco Core, SR-IOV VFs are exposed to pods in two modes:

* `netdevice` driver — a regular kernel network device visible inside the pod's network namespace, used for signaling and control-plane traffic.
* `vfio-pci` driver — a character device for DPDK applications running in user space, used for high-throughput user-plane (UPF) traffic.

The SR-IOV Network Operator provisions and manages SR-IOV CNI, the network device plugin, and other components of the SR-IOV stack.

You can read more about the SR-IOV operator {sriov-docs}[here].

[#metallb-operator]
== MetalLB Load Balancer

MetalLB is a load-balancer implementation for bare-metal Kubernetes clusters that uses standard routing protocols. It enables a Kubernetes Service to obtain an external IP address that is also added to the host network, making the service reachable from outside the cluster.

In Telco Core deployments MetalLB operates exclusively in BGP mode. Key resources used to configure it are:

* `IPAddressPool` — defines the pool of external IP addresses MetalLB can assign.
* `BGPPeer` — establishes a BGP peering session with a top-of-rack router, one per VLAN interface.
* `BGPAdvertisement` — controls which address pools are advertised to which BGP peers, enabling service traffic separation per VLAN.
* `BFDProfile` — configures Bidirectional Forwarding Detection for faster link-failure detection.

The MetalLB Operator deploys and manages the lifecycle of a MetalLB instance in the cluster.

You can read more about MetalLB {metallb-docs}[here].

[#nmstate-operator]
== NMState Operator

The Kubernetes NMState Operator provides a Kubernetes API for performing state-driven network configuration across cluster nodes using `NodeNetworkConfigurationPolicy` (NNCP) CRs. It is required in Telco Core clusters to configure the multi-layer networking topology defined by the RDS.

Typical configurations managed by the NMState Operator include:

* Linux bond interfaces in active-active LACP mode (IEEE 802.3ad).
* VLAN interfaces on top of the bond, used for cluster traffic, storage, and service separation.
* Static routes to ensure pod egress traffic is correctly steered through the appropriate VLAN gateway.
* MTU settings and promiscuous mode on secondary interfaces.
* SR-IOV VF interfaces (VLAN tagging, MTU) when VFs are used for host networking.

Initial networking configuration at installation time is applied via `NMStateConfig` content in the installation CRs. The NMState Operator is used for any subsequent Day 2 network updates.

You can read more about the NMState operator {nmstate-docs}[here].

[#numa-resources-operator]
== NUMA Resources Operator

Telco Core clusters are deployed on servers with multi-NUMA architectures. The default Kubernetes scheduler only considers the total free resources across an entire node, which can cause pods to be admitted to a node where no single NUMA node can satisfy the resource request, resulting in a local admission failure.

The NUMA Resources Operator (NROP) solves this by deploying a topology-aware scheduler that understands NUMA locality. All Telco Core clusters with multi-NUMA worker nodes are required to deploy NROP.

Key capabilities:

* Deploys a secondary topology-aware scheduler alongside the default Kubernetes scheduler.
* Works in conjunction with the `PerformanceProfile` CR to enforce NUMA-aligned CPU and memory allocation for Guaranteed QoS pods.
* Supports schedulable control planes — NROP is supported for use on control plane nodes.

You can read more about the NUMA Resources Operator {nrop-docs}[here].

[#cluster-logging-operator]
== Cluster Logging Operator

The Logging Operator provides APIs to control the collection and forwarding of logs from all pods and nodes in a cluster. This includes application logs (from regular pods), infrastructure logs (from system pods and node logs), and audit logs (special node logs with legal/security implications).

In Telco Core configurations, the reference design uses Kafka to ship audit and infrastructure logs to a remote archive for analysis. The Logging Operator starts, configures, monitors, and manages the Vector log collector components that perform these tasks.

NOTE: The reference configuration does not include shipping of application logs by default. Including application logs requires evaluating the application logging rate and allocating sufficient additional CPU resources to the reserved set.

[#storage-operator]
== Storage Operators

Telco Core applications require highly available persistent storage. The reference solution is OpenShift Data Foundation (ODF) deployed in _external mode_, where ODF connects to a dedicated Red Hat Ceph Storage cluster running outside the application OpenShift cluster.

External mode is the preferred storage configuration for Telco Core deployments for several reasons:

* The external Ceph Storage cluster can be shared across multiple OpenShift clusters deployed in the same region.
* OpenShift Container Platform and Ceph Storage can be updated independently, decoupling their lifecycle management.
* Storage and OpenShift platform operations responsibilities are cleanly separated, which is a typical requirement for telco core use cases.

ODF supports separation of storage traffic using secondary CNI networks. Storage network traffic must be isolated on a dedicated VLAN and must not be routable from other cluster networks.

You can read more about OpenShift Data Foundation {odf-docs}[here].

[#cert-manager-operator]
== Cert-Manager Operator

The cert-manager Operator for Red Hat OpenShift is a cluster-wide service that provides application certificate lifecycle management. The cert-manager Operator for Red Hat OpenShift allows you to integrate with external certificate authorities and provides certificate provisioning, renewal, and retirement.

You can read more about the Cert Manager Operator {cert-manager-docs}[here].
